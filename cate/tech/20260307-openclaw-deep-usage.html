OpenClaw 深度使用方案：从能用到好用

目标读者： 已入门 OpenClaw、希望深入掌握其核心能力的高级开发者与架构师

前言：为什么你需要深度掌握 OpenClaw？

OpenClaw（俗称"龙虾"）自 2025 年底发布以来，在 GitHub 上获得了超过 25 万星标，成为 2026 年最受欢迎的开源 AI 项目。但很多开发者停留在"能跑通"的阶段——配置好 WhatsApp、连上 Claude，开始聊天，然后止步于此。

这种使用方式只发挥了 OpenClaw 10% 的潜力。

真正的价值在于：它不是聊天机器人，而是一个 本地的智能体操作系统。理解了这一点，你才能构建出：

  • 能 7×24 小时自动执行任务的数字员工
  • 多智能体协同的 AI 团队，替代传统开发流程
  • 本地优先的数据主权与隐私保障体系

本文将系统性地拆解 OpenClaw 的三大核心模块：多通道架构、高级技巧、记忆系统，并基于生态演进给出趋势判断和创意应用场景。

一、OpenClaw 高级功能与技巧

1.1 多通道架构：消息的统一接入网关

OpenClaw 的核心设计之一是 Gateway（网关）中心化架构。所有通讯渠道——WhatsApp、Telegram、Discord、Slack、iMessage 等——通过统一的 WebSocket 控制平面（默认 ws://127.0.0.1:18789）接入。这意味着：

  • 一个 Gateway 进程服务所有渠道，会话和状态统一管理
  • 渠道适配器插件化，每个渠道通过标准接口注册（registerChannel）
  • 消息路由可编程，通过 bindings 规则实现精准分发

实战技巧：智能路由配置

不要让所有消息都进入同一个智能体。通过 openclaw.json 中的 bindings 字段，可以实现：

按渠道/账户路由（适合多场景隔离）：

{
  "bindings": [
    {
      "agentId": "work",
      "match": {
        "channel": "whatsapp",
        "accountId": "biz"
      }
    },
    {
      "agentId": "personal",
      "match": {
        "channel": "telegram"
      }
    }
  ]
}

按群组/用户路由（适合群聊机器人）：

{
  "bindings": [
    {
      "agentId": "dev-team",
      "match": {
        "channel": "discord",
        "peer": {
          "kind": "guild",
          "id": "897654321098765432"
        }
      }
    },
    {
      "agentId": "support",
      "match": {
        "channel": "whatsapp",
        "peer": {
          "kind": "dm",
          "id": "+15551234567"
        }
      }
    }
  ]
}

路由优先级（从高到低）：

  1. peer（精确匹配群/个人 ID）
  2. parentPeer（线程继承）
  3. guildId/teamId（服务器/团队 ID）
  4. accountId（账户 ID）
  5. 渠道级兜底
  6. 默认智能体

核心原则：具体规则在前，兜底规则在后，避免消息误分发。

高级特性：DM Scope 会话隔离

多用户场景下（开放 DM 给多人），默认的 "dmScope": "main" 会导致所有 DM 共享同一会话上下文，存在隐私泄露风险。

推荐配置：

{
  "session": {
    "dmScope": "per-channel-peer"
  }
}

每个渠道的每个用户独立会话，真正实现"一人一脑"。

1.2 节点系统：延伸到设备边缘的执行能力

OpenClaw 的 Nodes（节点） 是其分布式执行能力的核心。节点是运行在其他设备上的轻量级代理，通过 WebSocket 连接到 Gateway，扩展执行范围：

节点类型 能力 适用场景
macOS Node 系统命令、通知、屏幕录制 本地自动化、脚本执行
iOS Node 语音唤醒、Canvas、相机 移动端交互、拍照上传
Android Node Canvas、相机、屏幕捕捉 手机操作自动化
Headless Node 无头模式执行 云服务器任务执行

节点配对机制

节点配对（Pairing）是安全的关键：

# 在 Gateway 端生成配对码
openclaw pairing generate

# 在节点设备上完成配对
openclaw node pair --code <配对码>

配对后，节点会声明自己的能力（capabilities），Gateway 可以根据需要动态调用。

实战技巧：跨设备工作流

场景：在手机上拍照，桌面端自动处理并上传到云服务。

  1. iOS 节点通过相机拍照，保存到 Canvas
  2. 通过 nodes.invoke 调用桌面节点的处理脚本
  3. 桌面节点执行图像处理，上传到 S3/OSS

关键能力：节点不仅仅是远程执行，而是 分布式编排 的组成部分。

1.3 技能（Skills）：工作流的可复用单元

OpenClaw 中的 Skills 不是传统意义上的"工具"，而是 工作流编排方案。

技能 vs 工具的本质区别

特性 工具（Tools） 技能（Skills）
粒度 原子操作，一次只做一件事（读文件、发消息） 编排方案，告诉 AI 如何串联工具完成任务
实现 TypeScript/Python 代码函数 Markdown 文件（包含步骤、逻辑）
调用方式 AI 直接通过函数调用 AI 读取 Markdown 后理解执行流程
扩展性 每个工具独立开发 可组合多个工具形成完整业务流

类比：工具是厨房里的刀、锅、铲子——每个只做一件事。技能是菜谱——告诉你先切菜、再热油、然后翻炒、最后调味。

技能生态：ClawHub

截至 2026 年 3 月，ClawHub 技能市场 已有超过 10,000+ 社区技能。热门技能类别包括：

  • 办公自动化：日历管理、邮件处理、文档生成
  • 内容创作：文章撰写、视频脚本、图像生成
  • 开发辅助：代码审查、测试生成、文档更新
  • 系统集成：1Password、Notion、GitHub、Trello

安装技能：

openclaw install <skill-name>

技能会安装到 ~/.openclaw/skills/ 目录，AI 自动发现并加载。

实战技巧：开发自定义技能

创建一个技能只需要编写 SKILL.md 文件：

# 技能：日报生成

## 描述
自动收集今日工作内容，生成结构化日报并发送到 Slack。

## 使用场景
- 每天下班前自动生成日报
- 根据历史记录优化日报格式

## 步骤

1. 查询今日的日历事件和任务
   - 使用 `google_calendar_search` 获取今日日程
   - 使用 `todoist_list` 获取完成的任务

2. 整理工作内容
   - 按项目分组
   - 标注完成状态

3. 生成日报 Markdown
   - 使用标准化模板
   - 包含工作时长、产出物、问题记录

4. 发送到 Slack
   - 使用 `slack_send` 发送到 #daily-report 频道
   - @ 相关负责人

## 配置
- Slack 频道：#daily-report
- @ 负责人：@manager, @tech-lead

AI 会读取这个文件，理解步骤，然后自动执行。

1.4 记忆系统：三层架构与持久化

OpenClaw 的记忆系统不是简单的向量检索，而是一套 围绕智能体长期运行场景设计的完整记忆系统，支持：

  • 可沉淀：跨会话持久化
  • 可检索：高效召回
  • 可治理：人工干预和权限管理
  • 可审查：追溯来源

三层记忆架构

┌─────────────────────────────────────┐
│ 会话记忆（Session Memory）           │
│ 当前对话上下文，会话结束即消失        │
│ 容量受模型上下文窗口限制              │
└──────────────────┬──────────────────┘
                   │ 触发 compaction 时自动刷新
                   ▼
┌─────────────────────────────────────┐
│ 短期记忆（Daily Logs）               │
│ memory/yyyy-mm-dd.md                │
│ 每天一个文件，自动加载今天+昨天        │
└──────────────────┬──────────────────┘
                   │ 手动或自动整理
                   ▼
┌─────────────────────────────────────┐
│ 长期记忆（Memory.md）                │
│ 精选的持久信息                       │
│ 每次会话启动时自动加载                │
└─────────────────────────────────────┘

记忆文件详解

memory.md（长期记忆）：存储精选的、持久有效的信息。每次 AI 启动都会加载这个文件：

# 长期记忆

## 用户偏好
- 编程语言：TypeScript（不用分号）
- 框架：Next.js 14+ React 18
- 代码风格：函数式优先，避免 class

## 项目信息
- 当前项目：电商平台重构
- 部署：Vercel（前端）+ Railway（后端）

## 重要决策
- 2026-02-20：从 REST 迁移到 tRPC
- 2026-02-22：支付方案选 Stripe

user.md（用户画像）：专门描述"你是谁"：

# 用户画像

## 基本信息
- 姓名：老金 | 时区：Asia/Shanghai

## 技术偏好
- TypeScript（不用分号，单引号，2 空格缩进）
- React + Next.js | 后端：Node.js + Fastify

## 沟通偏好
- 喜欢直接，代码示例比文字解释有用
- 先给方案，再解释原因

开启 AutoLearn：AI 会在对话中自动捕捉偏好并更新 user.md。你说"改成箭头函数，我"，它会自动记住不喜欢 function 关键字。

记忆的工作原理

OpenClaw 使用 SQLite-Vec 做向量存储——基于 SQLite 的轻量向量数据库，零配置、单文件，支持语义搜索。

AI 搜索记忆时不是简单关键词匹配，而是语义理解。你问"我之前用什么支付方案"，即使记忆里写的是"选了 Stripe 作为支付服务商"，也能匹配到。

检索能力：

  • 混合搜索：关键词匹配 + 向量语义搜索综合排序
  • 时间衰减：越新的记忆权重越高
  • 去重：避免返回高度相似的内容
  • 查询扩展：自动扩展搜索词，提高召回率

Compaction：上下文压缩

对话越来越长，上下文窗口快满了怎么办？OpenClaw 会触发 compaction（压缩）。

压缩前，系统会先让 AI 把重要信息写入记忆文件，然后把早期消息压缩成摘要，释放空间继续对话：

正常对话：
[系统提示 | 记忆 | 消息1 | 消息2 | ... | 消息n]

接近上限：
[系统提示 | 记忆 | [摘要:消息1-50] | 消息51 | ... | 消息n]

你也可以手动触发：在对话中输入 /compact 命令。

1.5 多智能体协作：从单兵作战到团队协同

OpenClaw 原生支持多智能体协作，这是其"Agent Native"架构的核心体现。一个 Gateway 可以托管多个独立的智能体，每个智能体有：

  • 独立的 workspace（工作区）
  • 独立的 agentDir（状态目录，包含认证、会话存储）
  • 独立的会话上下文（除非明确启用，否则无交叉通信）
  • 独立的系统提示词（SOUL.md、IDENTITY.md）

什么时候需要拆分智能体？

信号一：记忆串线，场景混淆

  • 工作群聊中的 AI 突然提及私人私聊内容，开发任务中冒出写作相关的记忆。

信号二：系统提示词臃肿，上下文膨胀

  • 为了适配不同场景，给 AI 叠加了大量要求，系统提示词越来越长，每次对话都要消耗数万 token。

信号三：Compaction 互相阻塞，会话卡顿

  • 一个会话的 compaction 运行时，其他所有会话都会被阻塞，多个场景同时使用时，卡顿问题明显。

创建多个智能体

方式 A：命令行快速创建

openclaw agents add work
openclaw agents add writing
openclaw agents add personal

自动生成独立的 workspace、agentDir 和 sessions 目录。

方式 B：配置文件精细控制

{
  "agents": {
    "list": [
      {
        "id": "home",
        "default": true,
        "name": "生活助手",
        "workspace": "~/.openclaw/workspace-home"
      },
      {
        "id": "work",
        "name": "工作开发",
        "workspace": "~/.openclaw/workspace-work"
      }
    ]
  }
}

智能体间通信

主代理 → 子代理：

sessions_spawn({
  mode: "session",
  runtime: "subagent",
  label: "dev",
  cwd: "agents/dev-agent",
  task: "你是全栈开发，完成登录页面的实现。"
})

子代理 → 主代理：

  • 子代理只需正常输出，主代理通过 sessions_history 或自动事件接收其输出。

跨会话通信：

sessions_send(
  sessionKey: "agent:main:subagent:xxxx",
  message: "请验收上述代码，运行测试后给我结果。"
)

实战场景：三智能体写作团队

创建三个智能体：

  • outline-agent：擅长提纲，逻辑清晰
  • writer-agent：擅长叙述，文笔优美
  • editor-agent：严格校对，格式强迫症

工作流：

  1. 创建提纲 Agent
  2. 获取提纲，交给写手
  3. 获取文章，交给编辑

1.6 安全与隔离：Docker 沙箱与 9 层权限体系

Docker 沙箱隔离

OpenClaw 可以在 Docker 容器内运行工具 以减少影响范围。

沙箱模式：

{
  "agents": {
    "defaults": {
      "sandbox": {
        "mode": "non-main",  // off | non-main | all
        "scope": "session",  // session | agent | shared
        "workspaceAccess": "none"  // none | ro | rw
      }
    }
  }
}

模式说明：

  • off：不使用沙箱隔离
  • non-main：仅沙箱隔离非主会话（默认）
  • all：每个会话都在沙箱中运行

范围说明：

  • session：每个会话一个容器（默认）
  • agent：每个智能体一个容器
  • shared：所有沙箱会话共享一个容器

工作区访问：

  • none：工具看到 ~/.openclaw/sandboxes 下的沙箱工作区（默认）
  • ro：以只读方式挂载智能体工作区（禁用 write/edit）
  • rw：以读写方式挂载智能体工作区

9 层权限体系

OpenClaw 使用分层策略评估工具调用权限（按优先级排序）：

  1. Profile（当前激活的配置文件）
  2. Provider（LLM 提供商级别）
  3. Global（全局默认设置）
  4. Agent（特定智能体配置）
  5. Group（智能体分组策略）
  6. Sandbox（沙箱环境限制）
  7. Subagent（子智能体委托限制）
  8. Session（当前会话临时策略）
  9. Invocation（单次调用参数）

配置示例：

{
  "tools": {
    "policy": "default-deny",
    "allowedTools": ["Bash", "Read", "Write"],
    "blockedTools": ["Bash(sudo:*)"]
  }
}

核心原则：默认拒绝，白名单放行，危险操作需要人工审批。

1.7 MCP（Model Context Protocol）：AI 世界的 USB-C

OpenClaw 原生支持 MCP 协议——由 Anthropic 于 2024 年 11 月发起的开放标准协议，定义了 AI 应用（Host）和外部工具/数据源（Server）之间的统一通信接口。

MCP 的三大核心能力

能力 说明 示例
Resources 静态数据和内容，如文件、数据库记录 Google Drive 文件、Notion 页面
Prompts 预定义的提示词模板，可带参数填充 "翻译这段文本到 {language}"
Tools 可执行的函数，类似 OpenClaw 的 tools 数据库查询、API 调用

配置 MCP Server

在 openclaw.json 中声明 MCP Server：

{
  "mcpServers": {
    "notion": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-notion"],
      "env": {
        "NOTION_API_KEY": "${NOTION_API_KEY}",
        "NOTION_AUTH_TOKEN": "${NOTION_AUTH_TOKEN}"
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "POSTGRES_CONNECTION_STRING": "${POSTGRES_CONNECTION_STRING}"
      }
    }
  }
}

传输方式：

传输 协议 启动方式 适用场景
stdio JSON-RPC 2.0 over stdin/stdout OpenClaw 启动子进程 本地 CLI Server（最常用）
sse JSON-RPC 2.0 over Server-Sent Events Server 独立运行 远程 Server / Docker 容器内

MCP + Skill：黄金组合

MCP 连接外部服务，Skill 编排工作流。两者协作远大于任何一方单独使用。

示例：从 Notion 读取待办，调用 GitHub API 创建 Issue，完成后更新 Notion 状态。

1.8 ACP（Agent Client Protocol）：连接 IDE 的神经系统

ACP 是 OpenClaw 与 IDE（如 Zed、JetBrains）等工具通信的标准协议。它不是 REST 或 gRPC，而是基于 WebSocket 构建的 全双工、类型安全、支持状态同步 的通信协议。

为什么不用 REST 或 gRPC？

  • 长时运行（long-lived processes）：复杂的智能体任务可能持续数分钟，HTTP 的 request/response 模型容易超时
  • 流式思考（streaming thought）：用户希望看到智能体的思考过程像打字机一样逐字输出，需要服务器主动推送能力
  • 状态同步（state synchronization）：智能体的状态（正在运行的任务、消耗的 token）动态变化，客户端需要实时感知
  • 人机介入（human-in-the-loop）：当智能体想要执行危险操作时，需要暂停并向用户发起"审批请求"

使用 ACP

# 启动 ACP 桥接
openclaw acp --url wss://gateway-host:18789 --token <token>

# 连接到特定智能体
openclaw acp --session agent:main:main

审批流程

当智能体决定执行敏感操作时，触发审批流程：

  1. Request Approval：服务端发送 approval.request 事件
  2. User Interaction：客户端收到事件，弹出对话框
  3. Grant/Deny：用户点击后，客户端发送 approval.grant 或 approval.deny 请求
  4. Resume：服务端收到许可，runtime 解除挂起，继续执行

二、未来发展趋势分析

2.1 从"消耗体"到"增殖体"的范式转变

当前大多数 AI 智能体是 外部依赖型消耗体：每次 LLM 调用都消耗 Token，经济价值 = 创造的价值 - Token 成本。

OpenClaw 代表的未来方向是 内在增殖型生命体：

三大创新点：

  1. 代理自修改闭环：智能体可以自主修改自己的代码、配置、技能，实现自我进化
  2. 数字身份交易：soul.md 成为可交易的身份资产，经过训练的智能体可以在市场上流通
  3. 隐性算力向边缘回流：从云端推理转向本地模型（如 Ollama），降低长期成本

核心指标：单次 LLM 调用成本 < 其创造的经济价值。

2.2 企业级发行版：RedHat 之于 Linux

OpenClaw 作为开源项目，直接进入企业面临挑战：安全、合规、审计、SLA。

企业级发行版（如 Zeelin-Claw）正在涌现，提供：

  • 最小权限 + 审批流：严格的权限控制和操作审计
  • SOP 标准化：企业工作流程固化
  • 多生态解耦适配：与现有企业系统无缝集成
  • ISO 安全认证：满足合规要求

趋势预测：2026-2027 年，企业级发行版将成为企业采用 OpenClaw 的主流方式。

2.3 技能经济的兴起

OpenClaw 的技能市场正在形成 去中心化的 A2A（Agent-to-Agent）交易生态：

  • 技能作为经济原子：每个技能都是独立的价值单元，可以定价、交易
  • 行动遗传：卖的不是"时间"，而是"行动遗传"——经过训练的工作流
  • 知识-行动的本体融合：内容和行动融合成可复用的知识孢子

预测：到 2028 年，技能经济将催生 1.2 万亿美元影子 GDP。

2.4 跨设备协同的标准化

OpenClaw 的节点系统代表了 跨设备协同 的方向。未来，协议会标准化：

  • 设备能力声明：设备自动声明能力（相机、麦克风、屏幕）
  • 安全配对机制：设备间安全握手，避免中间人攻击
  • 统一控制平面：所有设备通过统一网关管理

预测：2027 年，跨设备协同协议会成为行业标准，类似蓝牙的设备发现与配对。

2.5 安全与隐私的演进

当前挑战：

  • 高权限风险：智能体可执行任意系统命令
  • 提示词注入：恶意文本可能导致越权操作
  • 公网暴露：数万个 OpenClaw 实例暴露在公网

未来方向：

  • 硬件隔离：专用 AI 安全芯片（如 Apple M 系列的 Secure Enclave）
  • 零信任架构：每个工具调用都需要实时审批
  • 联邦学习：本地模型训练，全局模型更新，数据不离开设备

3.1 一人公司的运营中枢

实现：

  • PM 智能体：需求拆解、任务分配
  • Dev 智能体：编码实现
  • QA 智能体：测试用例编写与执行
  • Reviewer 智能体：PR 评审

效果：一人团队产出团队级效率，毛利率达 60-80%（远超传统 30-50%）。

核心瓶颈从创始人时间变为智能体部署数量。

3.2 隐私优先的个人助理

场景：数据敏感型用户（医生、律师、金融从业者）的本地化 AI 助理。

实现：

  • 本地模型：使用 Ollama + Llama 3，完全离线运行
  • 加密存储：所有记忆文件使用 AES-256 加密
  • 严格权限：只允许特定设备接入

优势：

  • 数据从未离开设备
  • 即使断网也能工作
  • 符合 GDPR/HIPAA 等合规要求

3.3 跨平台内容创作流水线

场景：内容创作者（视频博主、播客主）的一键生成流水线。

实现：

  • 搜集智能体：监控特定主题的新闻、论文、社交媒体
  • 分析智能体：分析热点，生成选题
  • 创作智能体：撰写脚本、生成大纲
  • 制作智能体：生成视频、配音、剪辑
  • 发布智能体：多平台自动发布（YouTube、B站、抖音）

效果：从选题到发布全自动，创作者只需做创意决策。

3.4 企业内部知识管理系统

场景：大型企业的内部知识检索、问答、培训。

  • 智能问答：员工可以通过 Slack/Teams 自然语言提问
  • 培训生成：自动生成培训材料、考核题目

优势：

  • 知识实时更新
  • 检索效率提升 10 倍以上
  • 降低新人培训成本

3.5 智能家居控制中枢

结语：从"使用 AI"到"管理 AI 团队"

OpenClaw 代表了 AI 智能体从"对话"向"行动"演进的关键趋势。它不是一个聊天机器人，而是一个 本地优先、Agent 原生的智能体操作系统。

深度掌握 OpenClaw，核心在于理解：

  • Gateway 是中枢：统一控制平面，管理所有渠道、节点、智能体
  • 智能体是独立的：每个智能体有独立的 workspace、记忆、会话
  • 记忆是资产：不是简单的向量检索，而是可沉淀、可治理、可审查的记忆系统
  • 安全是底线：Docker 沙箱、9 层权限体系、审批流程缺一不可
  • 生态是杠杆：MCP、ACP、ClawHub 构成了强大的扩展生态

最终目标：你不是在"使用 AI"，而是在"管理 AI 团队"。从单兵作战到团队协同，从消耗体到增殖体，这就是 OpenClaw 的核心价值。

未来属于那些最先与自己的智能体达成共生的人。
